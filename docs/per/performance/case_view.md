# 营销活动压测面试 · 表达文字稿(完整版)

## 开场定位(30秒)

面试官您好,我想分享一个我之前深度参与的活动压测案例——某运营商的"全网移动云盘会员日"活动。这个活动有几个典型特征:每月会员日开启当天才能做任务(拉新、拉活、AI工具试用)来获取抽奖次数,奖品有库存限制。这类场景在大厂营销活动中非常常见,我围绕它完整走了一遍全链路压测的流程。

---

## 一、案例背景与特征(1分钟)

### 活动基本情况
这个活动的时间跨度是2025年7月到2027年7月,是个长期活动。核心玩法是会员日当天用户完成拉新、拉活、AI工具试用等任务来获取抽奖次数或积分,然后抽奖兑换奖品,奖品有库存限制。

### 核心数据指标
我这边拿到了活动的核心数据:
- **PV达到131万,UV是62.6万**,人均浏览深度约2.1
- **登录用户数46.9万,登录转化率75%**
- **中奖数15.4万次,中奖率32.8%**
- **领奖数7.8万,领奖率50.6%**

### 流量特征分析
这个活动有几个关键的技术特征需要特别注意:
1. **时间敏感性**:任务仅在会员日当天可做,流量时间分布高度偏斜,开启瞬间会有突增
2. **高并发写操作**:核心写接口集中在活动开启首日
3. **库存竞争**:奖品有库存限制,涉及扣减操作

---

## 二、技术风险识别(1.5分钟)

基于活动特性,我识别了六大核心风险点:

### 风险一:库存超卖(严重等级最高)
这是最关键的风险。如果使用"查库存→扣减"的非原子操作,或者缓存与数据库不一致,会导致奖品超发。我见过实际案例中库存变成负数的情况。

### 风险二:重复领取(幂等性问题)
网络超时导致用户重试,后端如果没有做幂等校验,同一个任务可能多次发放积分或奖品。

### 风险三:外部依赖故障
我们调用了AI工具(如文生图),这些外部依赖如果响应超时超过3秒,可能占满Tomcat线程池,导致级联故障。

### 风险四:缓存击穿/雪崩
热点奖品Key在Redis中失效后,大量请求直接打到数据库,这是经典的缓存问题。

### 风险五:数据库行锁竞争
更新用户积分时如果没有索引或锁等待超时,高并发下死锁率会上升。

### 风险六:日志监控被冲垮
500 TPS下如果每个请求都打印大量日志,可能撑爆磁盘IO和ES索引队列。

## 三、压测指标获取与计算(2分钟)

### 数据来源策略
我采用的是**"历史基线+业务预期+漏斗转化"**三层校准的方法:
- **APM埋点日志**(SkyWalking)权重50-60%,这是最精准的,包含实时调用量、RT、错误率
- **业务侧预测增长**权重20-30%
- **漏斗转化衰减**权重10-20%

### SkyWalking数据提取
SkyWalking提供了几个核心指标:
- **CPM**(Calls Per Minute):每分钟调用数
- **响应时间**:P50/P75/P90/P95/P99 RT
- **错误率**和**SLA**

我具体是这样操作的:
1. 先在Dashboard的Service Dashboard中查看活动当天全天的CPM折线图,定位高峰时段(比如10:05)
2. 然后把时间范围改为高峰时段(如10:00-10:15),进入Service→Endpoint List
3. 最后导出CSV文件,里面包含各接口的CPM和P99 RT

### QPS/TPS/并发数计算
核心公式有三个:
- **QPS = CPM ÷ 60**(读接口GET)
- **TPS = CPM ÷ 60**(写接口POST/PUT/DELETE)
- **并发数 = QPS × 平均响应时间(秒)**,这是Little定律

举个实际例子,在高峰时段10:00-10:10:
- **GET /api/user/info**:CPM 18,000,P99 RT 150ms → QPS 300,并发45
- **POST /api/lottery/draw**:CPM 6,000,P99 RT 300ms → TPS 100,并发30
- **POST /api/task/submit**:CPM 30,000,P99 RT 200ms → TPS 500,并发100

合计目标TPS约700-1000,实际达标850 TPS。

## 四、JMeter压测方案设计(2分钟)

### 整体策略
我采用的是**"梯度增压+混合场景"**策略,模拟真实用户路径:登录→做任务→抽奖。

### 核心元件配置

#### 线程组设置
- **主线程组**:使用Concurrency Thread Group,阶梯爬坡从100到1000 TPS,每阶段持续5分钟,逐步寻找系统拐点
- **预热线程组**:setUp Thread Group提前缓存登录态,预热系统

#### 数据准备
通过CSV Data Set Config准备50万测试用户数据(手机号+用户ID),模拟真实UV。

#### 流量配比控制
使用Throughput Controller控制流量配比:任务提交60%、抽奖30%、登录10%,这个配比是基于业务数据来的。

#### 幂等性保证
这是个关键点。我使用JSR223 PreProcessor生成幂等Token(UUID+时间戳+UserID的MD5),放入Header的Idempotent-Key字段,模拟防重提交。

#### 监控与断言
- 配置InfluxDB+Grafana实时监控TPS、RT百分位(P99)、错误率
- Response Assertion校验响应是否包含"success":true
- Duration Assertion设置核心接口RT≤500ms,登录接口RT≤1s

### 针对特定风险的压测策略
- **超卖测试**:使用Synchronizing Timer设定50线程同时释放,模拟50人同时抢最后一个库存,观察是否返回负数
- **AI接口超时**:HTTP Request单独设置连接超时1s、响应超时2s,配合Constant Throughput Timer在错误率>5%时强制降级
- **缓存穿透**:固定热点奖品ID,清除Redis缓存后并发压测,观察数据库的QPS和连接数飙升情况

## 五、面试高频问题答疑

### Q1:压测链路业务配比,当只有ELK+Nginx时如何获取?

这个问题很好。当没有APM时,Nginx access.log是最基础的数据来源。

**Nginx日志提取接口PV分布**,我常用的生产级命令是:
```bash
# 过滤静态资源,只看业务接口
cat access.log | grep -v -E '\.(css|js|png|jpg|gif|ico)$' \
  | awk '{print $7}' | sort | uniq -c | sort -rn | head -20

# 按高峰时段切片
grep "15/Jul/2025:10:0[0-9]" access.log \
  | awk '{print $7}' | sort | uniq -c | sort -rn | head -20
```

**ELK提取业务转化率**方面,典型查询是:
```
# 查询中奖日志
message: "lottery_success" AND activity: "会员日"
```

**整体流程**是:
1. Nginx日志→各接口PV比例(登录10%、任务提交60%、抽奖30%)
2. ELK日志→业务转化率(中奖率32.83%、领奖率50.62%)
3. 业务数据库→用户漏斗数据
4. 三者结合→校准压测脚本的业务配比

我的经验是:当只有Nginx+ELK时,先用Shell命令或GoAccess从Nginx日志提取接口PV分布,再用ELK查业务埋点日志获取转化率。虽然没有APM的链路追踪,但组合使用仍能得到合理的压测配比。

### Q2:如何评估并发用户数?压测机台数怎么算?(QPS=1万,RT=500ms)

这个我分两步计算。

**第一步:计算总并发线程数(VU)**
根据Little定律:并发用户数 = QPS × 平均响应时间(秒)
代入:10,000 QPS × 0.5s = **5,000并发线程(VU)**

**第二步:计算所需压测机台数**

关于单台施压机上限,我的经验是:
- JMeter每个线程(VU)默认占用约1MB堆内存
- 8C16G施压机,分配-Xmx12G,理论上限12,000线程
- **生产经验值**:单台安全并发控制在1,500~2,000 VU(超过则GC频繁,数据失真)

**计算公式**:
- 所需台数 = 5,000 ÷ 1,500 ≈ **3.3台**
- **最终建议**:准备4台施压机(3主力+1冗余),确保施压机带宽>被压机带宽

### Q3:压测时TPS上不去但CPU/内存不高的排查方向?Arthas何时上场?

核心结论是:**TPS上不去但CPU/内存空闲→瓶颈不在应用计算或堆内存,而是卡在"IO等待"或"外部协调"**。

**基础排查方向**(按优先级):
1. **数据库/Redis连接池**:查看DataSource活跃连接数是否等于maximumPoolSize,大量线程在等待连接
2. **下游依赖超时**:检查调用第三方接口的响应超时配置,线程堆栈是否有大量SocketTimeout
3. **网卡带宽打满**:用iftop/nload查看网卡流量是否达到内网限速
4. **OS内核限制**:检查net.ipv4.ip_local_port_range是否耗尽,TIME_WAIT堆积
5. **分布式锁竞争**:抽奖扣库存用Redisson或select...for update,大量线程在自旋等待锁释放
6. **GC停顿**:查看GC日志,检查是否频繁发生STW

**关于Arthas**,当jstack看不出明显锁等待时,我会果断用Arthas。三个杀手级场景:

1. **定位外部接口超时根因**:用`trace com.xxx.TaskService submitTask`动态打印方法内每一步的精确耗时。如果发现900ms耗费在HttpClient.execute(),而业务代码仅用1ms→秒级定位网络IO超时

2. **线上临时开启DEBUG日志**:用`logger --name ROOT --level debug`,怀疑分布式锁tryLock在空转但线上INFO级别看不到。实时改为DEBUG观察等待时间,验证完再改回,免重启

3. **反编译验证线上代码版本**:用`jad --source-only com.xxx.StockService`,怀疑部署包不是最新代码。直接反编译JVM内存中的字节码,如果发现确实缺少判空逻辑→实锤截图,终结扯皮

我的经验是:当`top -H`和`jstack`看不出明显锁等待时,Arthas的`watch`命令能让我在不停服的情况下完成"手术刀式"的精准诊断。

### Q4:压测有没有遇到什么瓶颈,做了什么优化?

这个问题非常关键。压测过程中确实遇到了多个瓶颈,主要集中在**数据库连接池耗尽、缓存穿透导致DB压力、分布式锁竞争加剧**三个方面。我逐一展开:

#### 瓶颈一:数据库连接池耗尽

**问题现象是这样的**:压测到500 TPS时,错误率突然飙升至15%,应用日志大量报错HikariPool连接超时,通过jstack发现大量线程在等待连接。

**根因分析**:HikariCP默认配置maximumPoolSize只有10,远低于实际需求。每个抽奖请求涉及多表操作(用户积分扣减、库存扣减、中奖记录写入),平均耗时200ms。500 TPS下,理论连接需求 = 500 × 0.2 = 100个连接。

**优化方案**:我把maximumPoolSize从10调整为100,同时设置最小空闲连接数20,连接超时从30秒改为10秒。优化后错误率从15%降至0.02%,连接池利用率从95%降至60%,支持TPS从500提升到850。

#### 瓶颈二:热点奖品缓存穿透导致DB压力激增

**问题现象**:压测300 TPS持续10分钟后,数据库CPU飙升至85%,Redis命中率从95%降至60%。慢查询日志出现大量热点奖品的查询语句。

**根因分析**:热点奖品ID(如iPhone、现金红包)被大量并发请求,Redis中的库存Key设置TTL=10分钟,压测期间频繁过期导致缓存穿透,形成"缓存雪崩"。

**优化方案**:我采用了两种策略。第一种是热点Key永不过期+异步刷新,设置长TTL并启动后台线程每5分钟从数据库同步。第二种是使用Redis分布式锁防止缓存击穿,双重检查锁重建缓存。

**优化效果**:Redis命中率从60%回升至95%,DB CPU从85%降至35%,连接数稳定在80左右。

#### 瓶颈三:分布式锁竞争导致响应时间抖动

**问题现象**:压测600 TPS时,P99 RT合格(480ms),但P999 RT飙升至3.5秒,Grafana监控显示RT出现明显毛刺。

**根因分析**:抽奖扣库存使用Redisson.tryLock(),高并发下大量线程在自旋等待锁释放,锁持有时间过长导致等待队列堆积。

**优化方案**:我采用了三种方案。第一种是锁粒度优化,从全局锁改为按奖品ID的分段锁,进一步优化为按奖品ID+用户ID分锁。第二种是使用Redis Lua脚本原子扣减,无需分布式锁,扣减成功后异步写入DB。第三种是引入队列削峰,抽奖请求先入队,后台消费者批量处理。

**优化效果**:P999 RT从3.5秒降至600ms,锁等待超时错误从20%降至0,支持TPS从600提升到850。

#### 其他优化措施

除了这三个主要瓶颈,我还做了几个其他优化:日志优化方面,生产环境禁用DEBUG日志,INFO日志改为异步写入,磁盘IO从80%降至30%。索引优化方面,为用户积分和奖品库存表添加唯一索引,查询RT从200ms降至50ms。批量处理方面,中奖记录从单条插入改为批量插入,DB写入TPS从200提升到800。预热策略方面,压测前预热热点奖品缓存和用户积分缓存,冷启动RT从2秒降至500ms。

我的体会是:**压测最大的价值不是得出一个数字,而是发现瓶颈并验证优化效果**。我们这次压测从500 TPS提升到850 TPS,过程中解决了数据库连接池耗尽、缓存穿透、分布式锁竞争三个关键瓶颈,每个瓶颈的优化都是基于真实数据分析和根因定位,而不是盲目调参。
