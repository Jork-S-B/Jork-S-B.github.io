## 2022计酬redis优化案例

### 计酬引擎 CPU 性能调优实战

**案例背景**：在“广东移动渠道费用管理系统”的年度佣金回溯结算（Recalculation）任务中，系统需要对全省 500+ 核心渠道在过去一年的业务量进行重新计酬核算。

#### 1. 问题描述 (Problem)
在 K8s 集群中启动计酬批次任务后，负责结算的服务 Pod CPU 瞬间飙升至 100% 甚至触发 OOM 导致重启。

*   **现象**：`kubectl top pods` 显示该节点 CPU 持续满载，内存占用逼近 Limit 线。
*   **性能数据**：单次 500 个渠道的结算任务耗时超过 2 小时，远超业务要求的 15 分钟。

#### 2. 排查思路与工具 (Troubleshooting)

**第一步：全链路状态观察**

*   **K8s 监控**：通过 `kubectl logs -f [pod_name]` 观察到大量针对 Oracle 的重复 SQL 查询日志。
*   **应用诊断**：使用 **Arthas** 工具挂载到容器进程。

    *   执行 `thread -n 3` 发现 CPU 主要消耗在 **Groovy 规则解析**和 **BigDecimal 高精度计算**上。
    *   执行 `monitor -c 5 [ClassName] [MethodName]` 统计结算主方法耗时，发现单次结算内部循环了上千次。

**第二步：定位数据库 IO 瓶颈**

*   **分析**：计酬逻辑采用“滑动窗口”模式（如：计算当日佣金需要回溯前 30 天的业务达成率）。
*   **技术手段**：在 AOP 切面中统计 `JdbcTemplate` 执行次数。
*   **排查结果**：500 渠道 × 300 结算天数 = 150,000 次数据库 IO 交互。累计网络延迟和结果集封装耗时占总时长的 70%。

**第三步：Redis 缓存失效根因定位**

*   **分析**：原本设计了 Redis 缓存渠道基础数据，但监控显示命中率为 0。
*   **根因定位**：通过 Arthas 的 `watch` 命令观察缓存 Key 生成逻辑。

    ```java
    // 原始错误代码逻辑
    String cacheKey = "channel:rule:" + channelId + ":" + startDate + ":" + endDate;
    // 现象：由于 startDate 和 endDate 随结算日期每天滑动，导致缓存 Key 永远无法复用。
    ```

#### 3. 解决方案 (Solution)

**方案一：预加载机制 (Bulk Loading)**

*   **思路**：将“单渠道单日查询”改为“全量数据批量预加载”。
*   **实现**：在结算开始前，一次性从 Oracle 批量读取 500 渠道全年的结算因子数据（Hadoop 预处理后的回流数据），存储在本地 **Map 容器**中作为临时快照，实现“零 IO”计算。

**方案二：优化缓存 Key 设计与策略**

*   **实现**：修改 Redis 存储策略，不再以日期区间为 Key，而是以 `channelId` 为 Key 存储全量序列。
*   **代码改写（Java 示例）**：
    ```java
    // 优化后：按渠道 ID 缓存全量 List，在内存中进行 Stream 切片
    public List<Metric> getMetrics(String channelId, LocalDate start, LocalDate end) {
        List<Metric> allData = localCache.get(channelId); 
        return allData.stream()
            .filter(m -> !m.getDate().isBefore(start) && !m.getDate().isAfter(end))
            .collect(Collectors.toList());
    }
    ```

**方案三：Groovy 脚本编译优化**

*   **思路**：Groovy 脚本重复解析非常耗 CPU。
*   **实现**：引入 `GroovyClassLoader` 对规则脚本进行**编译预缓存**，避免在循环内部重复调用 `new GroovyShell().parse()`。同时将 `BigDecimal` 的标度转换逻辑移出循环体。

#### 4. 验证效果 (Result)

通过模拟 500 个渠道结算任务进行压测，对比数据如下：

| 指标 | 优化前 (Original) | 优化后 (Optimized) | 提升 |
| :--- | :--- | :--- | :--- |
| **数据库查询次数** | 150,000+ 次 | 2 次 (日线/周线预加载) | **99.9% 降低** |
| **计酬引擎 CPU 占用** | 100% (Pod 频繁限流) | 25% ~ 35% (平稳) | **大幅优化** |
| **500 渠道结算总耗时** | 7,200 秒 (2 小时) | 180 秒 (3 分钟) | **性能提升 40 倍** |
| **缓存命中率** | 0% | 100% (预加载模式) | **显著提升** |

## Arthas 命令详解

### 1. `dashboard` —— 实时全景监控
*   **功能**：类似于 Linux 的 `top` 命令，一屏展示 JVM 的 CPU、内存、GC、线程数等核心指标。
*   **在本案例中的作用**：
    *   第一眼确认 **CPU 占用率**（确认是用户态计算消耗还是内核态 I/O 等待）。
    *   观察 **GC 情况**（排除是否因内存泄漏导致频繁 Full GC 引起的 CPU 飙升）。
*   **面试表述**：*“我首先通过 `dashboard` 观察到 CPU 占用达到 95% 以上，且内存回收处于正常范围，排除了频繁 GC 导致的虚高，确定瓶颈在业务逻辑或 I/O。”*

---

### 2. `thread -n [count]` —— 定位最忙线程
*   **功能**：列出 CPU 占用率最高的前 N 个线程，并打印其堆栈信息。
*   **在本案例中的作用**：
    *   **命令**：`thread -n 3`
    *   **定位结果**：发现最忙的线程栈指向了 `groovy.lang.GroovyClassLoader.parseClass` 或 `oracle.jdbc.driver` 的等待。
*   **面试表述**：*“执行 `thread -n 3` 后，我直接抓到了‘凶手’：大量线程停留在 Groovy 脚本解析和数据库结果集封装上，这说明代码中存在高频的脚本编译动作和多余的数据库交互。”*

---

### 3. `watch` —— 动态入参/出参监测（关键命令）
*   **功能**：在不重启应用的情况下，实时查看某个方法的**入参、出参、异常、甚至类成员变量**。
*   **在本案例中的作用**：
    *   **命令**：`watch com.gd.settle.CacheService getCache "{params, returnObj}" -x 2`
    *   **定位结果**：通过该命令发现，明明入参是同一个渠道 ID，但生成的 `cacheKey` 每次都带有一个精确到毫秒的时间戳（或动态日期），导致 `returnObj` 永远为 `null`。
*   **面试表述**：*“为了查明缓存命中率为 0 的原因，我使用 `watch` 命令监控了缓存服务类。结果发现 Key 的生成逻辑中混入了动态滑动日期，导致每次查询的 Key 都是全新的，从而证实了缓存击穿的假设。”*

---

### 4. `trace` —— 方法调用链路耗时分析
*   **功能**：追踪方法内部各个子方法的调用时长。
*   **在本案例中的作用**：
    *   **命令**：`trace com.gd.settle.EngineService calculateTotalAmount`
    *   **定位结果**：显示单次计算耗时 200ms，其中 `queryFromDB` 占据了 180ms。
*   **面试表述**：*“通过 `trace` 逐层拆解计算方法，我量化了性能瓶颈：70% 以上的时间都消耗在 SQL 网络开销上，剩下的 20% 消耗在 Groovy 脚本的重复解析上。这为后续‘预加载’优化方案提供了数据支撑。”*

---

### 5. `monitor` —— 方法执行统计
*   **功能**：每隔一段时间（默认 120 秒）统计方法的调用次数、平均响应时间、成功率等。
*   **在本案例中的作用**：
    *   **命令**：`monitor -c 5 com.gd.settle.Engine calculate`（每 5 秒统计一次）
    *   **定位结果**：发现该核心计算方法在 1 分钟内被调用了数万次。
*   **面试表述**：*“我利用 `monitor` 统计了核心结算方法的 QPS，发现对于 500 个渠道的任务，内部循环计算调用了 15 万次。这让我意识到必须将‘循环查询’重构为‘批量处理’。”*

---

### 6. `profiler` —— 生成火焰图 (Flame Graph)
*   **功能**：采样生成 CPU 火焰图，直观展示哪些方法占用 CPU 最久。
*   **在本案例中的作用**：
    *   **命令**：`profiler start` -> `profiler stop --format html`
    *   **定位结果**：火焰图上“最宽”的平顶块是 Groovy 和 JDBC 相关的类。
*   **面试表述**：*“为了给研发提供更直观的证据，我生成了火焰图。图中显示 Groovy 脚本解析和数据库连接池获取占据了最大的横向比例，这让团队迅速达成了一致优化方案。”*

---

### 💡 Arthas 进阶技巧（测试加分项）

*   **热更新代码 (mc/redefine)**：如果在测试环境发现代码有个小 Bug，可以通过 `mc` (Memory Compiler) 编译源码，然后用 `redefine` 实时加载到运行中的 JVM 里，**无需重启容器**即可验证修复效果。
    *   *场景：* 修改缓存 Key 生成逻辑，立即验证 CPU 是否下降。
*   **反编译代码 (jad)**：如果你怀疑生产环境代码和 Git 分支代码不一致，可以用 `jad` 反编译 Pod 里的 `.class` 文件。
    *   *场景：* “为什么我改了 Bug 还是报错？” -> `jad com.xxx.Service` 发现 Pod 里跑的还是旧代码。

### 总结：我的排查路径
1.  **`dashboard`** 看整体。
2.  **`thread -n 3`** 找线程。
3.  **`trace`** 定位具体哪个子方法慢。
4.  **`watch`** 看数据（入参/Key）是否异常。
5.  **`profiler`** 出报告。
