# 202405酬金压测

性能压测与调优：使用 JMeter 对核心酬金结算接口进行阶梯加压，结合 Prometheus+Grafana 实时监控系统指标。

## 压测发现：

- 当并发数升至 400 时，TPS 达到峰值 500，但平均响应时间已升至 1.2s，P90 响应时间达 2.5s；
- 继续增加并发至 500，TPS 反而下降至 420，平均响应时间飙升至 2.2s，P90 高达 4.5s，系统出现严重瓶颈。

（TPS 先升后平，达到瓶颈）

## 第一步：全局监控，确定瓶颈层

此时通过Grafana观察系统资源指标：

* CPU使用率：在并发500时，CPU接近100%（user高，sys正常），说明计算密集型或锁竞争。
* 内存使用：平稳，无OOM，排除内存泄漏。

结论：瓶颈可能在应用层或数据库层，且与计算相关。

## 第二步：深入数据库层

* 查看数据库监控（MySQL的Prometheus exporter）：

连接数：达到最大连接数，大量连接处于“Sending data”状态。
慢查询数：突增，每分钟慢查询从0飙升至数百。

* 启用慢查询日志（MySQL特有），抓取耗时SQL：

=== "永久生效"

    ```shell
    # /etc/my.cnf
    log_output = table  # 将慢查询日志保存到表中
    slow_query_log = 1  # 开启慢查询
    long_query_time = 1  # 慢查询超时时间为1秒
    max_connections = 512  # 最大连接数，要求适中
    ```

=== "临时、重启失效"

    ```sql
    -- 开启慢查询日志（排查性能瓶颈必备）
    -- SET GLOBAL slow_query_log = 'ON';
    -- SET GLOBAL long_query_time = 1;  -- 超过1秒记录
    -- SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

    -- 查看是否开启
    SHOW VARIABLES LIKE 'slow_query_log';
    SHOW VARIABLES LIKE 'long_query_time';
    ```

通过慢查询日志，定位发现，核心SQL存在性能问题：

=== "使用函数导致索引失效"

    ```sql
    -- 差：WHERE DATE(create_time) = '2024-01-01'
    -- 好：WHERE create_time >= '2024-01-01' AND create_time < '2024-01-02'
    ```

=== "使用模糊查询导致索引失效"

    ```sql
    -- 差：WHERE channel_type = LIKE '%keyword%'
    -- 通过前端页面控制输入，如文本框改为下拉选择，精确匹配
    ```

=== "NOT IN 子查询"

    ```sql
    -- 差：WHERE channel_id NOT IN (SELECT channel_id FROM ...)
    -- 好：WHERE NOT EXISTS (SELECT 1 FROM ... WHERE ...)
    -- 注意：NOT IN 子查询含 NULL 时结果永远为空！NOT EXISTS 无此问题
    ```

=== "大表直接关联查询"

    ```sql
    SELECT f.*, bf.factor_value, bf.factor_weight
    FROM channel_fee_raw f
    INNER JOIN business_factor bf
        ON f.channel_id = bf.channel_id AND f.batch_date = bf.factor_date
    WHERE f.batch_date = '2024-01-01';
    -- EXPLAIN 可能显示：f 表 ALL 扫描 150万行，每行去 bf 表查找

    -- 原因分析
    -- 1. MySQL 优化器可能选错驱动表（应该小表驱动大表）
    -- 2. JOIN 条件无索引 → Nested Loop 每次都全表扫描被驱动表
    -- 3. WHERE 过滤在 JOIN 之后，数据量已膨胀

    -- 优化
    -- Step 1: 确认索引存在
    CREATE INDEX idx_factor_channel_date ON business_factor(channel_id, factor_date);

    -- Step 2: 用 STRAIGHT_JOIN 强制驱动表顺序（小表在前）
    SELECT f.*, bf.factor_value, bf.factor_weight
    FROM (
        SELECT * FROM channel_fee_raw WHERE batch_date = '2024-01-01'
    ) f
    STRAIGHT_JOIN business_factor bf
        ON f.channel_id = bf.channel_id AND f.batch_date = bf.factor_date;

    -- 更优方案：先通过子查询/CTE/聚合，过滤缩小数据量，确保被驱动表（join关联字段）有覆盖索引
    -- 必要时用 STRAIGHT_JOIN 强制驱动表顺序（小表驱动大表）
    SELECT f.*, bf.factor_value, bf.factor_weight
    FROM (
        SELECT id, channel_id, batch_date, fee_type, fee_amount
        FROM channel_fee_raw
        WHERE batch_date = '2024-01-01'
    ) f
    INNER JOIN business_factor bf
        ON f.channel_id = bf.channel_id AND f.batch_date = bf.factor_date;

    --- CTE: WITH 子句中定义的临时结果集

    ```

## 优化

推动开发优化索引结构及上述性能问题，优化后再次压测：

- 并发 1000 时，TPS 达到 1850，平均响应时间 500ms，P90 900ms；
- 并发 2000 时，TPS 稳定在 2000，平均响应时间 1.0s，P90 1.8s，系统吞吐量提升 3 倍，为上线提供了可靠保障。

## 执行计划

EXPLAIN ANALYZE + SQL

关键字段说明：

- rows = 10000 表示扫描大约10000行数据。
- possible_keys = NULL 表示没有合适的索引可用。

### type

- ALL：全表扫描（最差，必须优化）
- index：全索引扫描（比ALL好一点）
- range：范围扫描（BETWEEN、>、<、IN）
- ref：索引等值查找（非唯一索引）
- eq_ref：唯一索引等值查找（JOIN时理想状态）
- const：常量查找（主键/唯一索引等值，最快）
- NULL：无需访问表（如 SELECT 1+1）

### Extra

Using index：覆盖索引，无需回表（好）
Using filesort：额外排序（差，需优化 ORDER BY）
Using temporary：使用临时表（差，需优化 GROUP BY/DISTINCT）
