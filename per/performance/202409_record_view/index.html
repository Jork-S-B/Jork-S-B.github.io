    广告平台压测面试 · 表达文字稿(完整版)

    开场定位(30秒)

    面试官您好,我想分享一个我主导的广告平台压测案例。这个案例的背景是我们的广告平台出现了Redis内存占用持续上涨突破90%、核心接口RT明显抖动的预警。我负责了从问题定位、压测设计、瓶颈优化到最终上线的完整闭环。这个项目成功支撑了2024年9月期间的高峰流量,线上实现0报警。

    一、案例背景与问题定位(1分钟)

    问题背景

    我们的广告平台是一个存量系统,主要问题有两个: - Redis内存压力:内存占用持续上涨突破90%,接近爆满 - 接口性能抖动:核心接口RT出现明显波动,通过nginx-prometheus-exporter监控发现

    数据量级

    • 用户标签库:约5000万用户画像数据
    • 广告规则库:核心匹配规则约500+条,MySQL存储
    • 日常流量:DAU约12万,日常峰值TPS约1200-1500
    • 容量目标:需要支撑3-5倍冗余,即目标TPS需达到5000以上

    初步定位

    我首先根据报错日志排除了连接数问题,然后排查了慢查询日志(阈值100ms),发现了两个关键问题: 1. 存在个别超过200ms的慢查询 2. 某个大key:存储了广告位所有用户标签,value大小约2MB

    初步定位:慢查询+大key可能是导致Redis内存压力增大、接口耗时升高的原因。

    二、压测策略与设计(1.5分钟)

    测试环境设计

    我设计了两轮压测:

    第一轮:测试环境尝试复现 - 配置:8C 16G × 2节点(模拟生产50%的处理能力) - 目的:初步探测代码性能拐点,验证调优方案的有效性 - 策略:梯度加压,从500并发开始,每5分钟增加500,直到系统出现明显性能拐点

    第二轮:灰度环境压测 - 配置:16C 32G × 4节点 - 集群规模:移动云集群部署,Redis采用8G主从集群模式 - 时间:凌晨低峰小并发加压压测

    脚本设计要点

    在脚本设计过程中,我遇到了几个技术挑战:

    挑战一:加密参数处理 开发提供了接口文档和AESUtil.jar包。最初我使用BeanShell后置处理器对随机号码进行加密,但运行报错。分析后发现使用了lang3第三方工具包。

    解决方案: - 弃用BeanShell,改用JSR223(Groovy):BeanShell性能极差,在高并发下会频繁触发Full GC。Groovy具有编译缓存机制,性能接近原生Java - 依赖管理:将AESUtil.jar和commons-lang3.jar放入JMeter的lib/ext目录

    挑战二:加密性能影响 为避免加密影响机器CPU消耗和GC压力,我根据加解密方法,预生成了一批加密后的号码在CSV文件中参数化。

    分布式压测设置

    我配置了分布式压测环境,关键步骤包括: 1. 设置堆内存配置(通常为物理内存的50%) 2. 配置内网IP,确保主机和从机可以互相识别和连接 3. CSV参数化文件分发到执行机 4. master和slave关闭SSL认证 5. 设置mode=StrippedBatch,剥离响应数据只返回度量值,大幅降低Master的带宽压力

    我还编写了shell脚本,创建以时间戳命名的独立结果目录、参数输入并执行分布式无界面压测、自动生成可视化HTML报告。

    三、压测过程与结果(1分钟)

    压测过程分为三个阶段:

    压测阶段 并发线程 吞吐量(TPS) 平均RT P95 RT 错误率 结论
    优化前(生产) 5000 4800 310ms 1.8s 0.5% P95抖动剧烈,存在大Key阻塞
    极限探测(生产) 10000 9641 682ms 1.3s 0.03% 触达物理极限,CPU负载85%
    最终调优后 5000 5200 60ms 150ms 0% 性能大幅提升,余量充足

    通过压测,我发现了一个关键现象:随着并发量上升,内存几乎呈线性增长且不回落。

    四、瓶颈定位与优化(2分钟)

    瓶颈一:大Key问题

    定位过程: 我使用redis-cli --bigkeys分析,并配合Arthas追踪业务逻辑代码。

    根因分析: 发现系统在记录"用户广告曝光历史"时,使用了String结构且未设置合适的TTL。在高并发下,每秒产生数万个过期时间极长的Key。当内存触达maxmemory后,触发了allkeys-lru频繁逐出有效热数据,导致缓存命中率从95%跌至60%,大量请求穿透到数据库。

    优化方案: 我建议开发将数据结构优化为Hash(减少Key数量),并严格根据业务周期设置TTL滑动过期。

    深入定位: 不仅仅是bigkeys发现2MB,我还通过redis-cli --stat观察到,在大Key读取时,内网带宽出现明显的入流量激增,这确认了瓶颈在于I/O传输开销和应用端的反序列化压力。

    优化原理: Hash结构在字段较少时使用ziplist压缩列表,内存极其紧凑;且通过HGET仅取所需字段,将单次传输的数据量从2MB降到了20KB以内。

    优化效果: 压测显示内存水位稳定在65%左右,不再持续上涨。

    瓶颈二:同步调用外部风控API导致的响应慢

    定位过程: 我使用Arthas的trace命令追踪了多个Handler的doJudge方法,发现外部风控接口响应慢,业务需要拿到结果再判断是否展示广告。

    优化方案: 非强关联的接口通过异步进行优化,最终选择了线程池异步任务的方式。同时引入Sentinel进行降级,当风控API响应超过500ms时,自动熔断10秒,系统自动下发"通用兜底广告",优先保证核心业务不被外部依赖拖垮。

    优化效果: 核心接口P99从800ms降至150ms以内。

    五、面试高频问题答疑

    Q1:性能测试中如何评估并发用户数?

    这个问题很关键。我分两个场景来解答:

    集合点并发

    定义是所有虚拟用户在集合点等待,同时发起请求,模拟瞬时高并发场景。适用场景包括秒杀、抢购活动、限时优惠活动等。

    评估方法有三种:

    1. 基于历史峰值数据: 

      集合点并发数 = 历史峰值在线用户数 × 业务增长系数(1.5-2倍)
例如:去年双11秒杀峰值在线用户5万,今年预估增长30%
集合点并发数 = 50000 × 1.3 = 65000个虚拟用户

    2. 基于业务预估: 

      集合点并发数 = 活动总参与人数 × 参与率 × 同时抢购率
例如:活动宣传覆盖100万人,预计参与率20%,同时抢购率10%
集合点并发数 = 1000000 × 0.2 × 0.1 = 20000个虚拟用户

    3. 基于系统容量上限: 

      集合点并发数 = 目标TPS × 平均响应时间 × 安全系数(1.2-1.5)
例如:目标支撑10000 TPS,平均响应时间200ms
集合点并发数 = 10000 × 0.2 × 1.3 = 2600个虚拟用户

    线程并发

    定义是在某个时间段内持续发压的虚拟用户数,每个线程完成一次请求后立即发起下一次。适用场景包括常规业务压测、稳定性测试、容量规划等。

    评估方法使用Little's Law(利特尔法则): 

    并发用户数 = QPS × 平均响应时间

    实际案例(本次压测): - 目标TPS:5000(用户级别的完整业务流程) - 实际QPS:约15000-20000(系统内部的请求总数,包括Redis、MySQL、风控API) - 实际P99响应时间:150ms - 理论并发用户数 = 15000 × 0.15 = 2250个(基于QPS) - 实际设置:1000个线程(考虑波动和余量)

    Q2:假设压测时TPS上不去但CPU和内存都不高,你会从哪些方向排查?

    这个问题非常典型。我会从以下几个方向排查:

    第一,数据库层面(最可能): - 行锁竞争:当压测使用同一账户ID,数据库会串行化行锁,TPS上不去,而CPU/内存正常 - 排查:show engine innodb status查看锁等待 - 解决:改造为批量扣减+账户分片,或用Redis原子操作+异步落库

    • 数据库连接池:连接池默认最大连接数可能只有10-20,压测时连接被占满,线程等待
    • 排查:监控active connections、waiting threads
    • 解决:适当增大连接池,但注意数据库自身连接数上限

    第二,服务端线程池配置: - Tomcat/Undertow的maxThreads较小(如200),或acceptCount队列满了导致丢包 - 排查:查看服务端日志是否有"Thread pool exhausted"或拒绝连接 - 解决:增大maxThreads,同时结合压测并发数调整

    第三,压测机器及脚本问题: - 网络带宽:虽然CPU不高,但若响应体过大,千兆网卡可能打满 - 排查:iftop或nload观察压测机出网带宽 - 端口耗尽:短压测时,TIME_WAIT状态占满端口范围 - 排查:netstat -an | grep TIME_WAIT | wc -l - 解决:调整net.ipv4.ip_local_port_range和tcp_tw_reuse

    第四,外部依赖/下游接口: - 若调用风控、积分等外部服务,且未设置合理超时+熔断,会导致线程阻塞 - 排查:用Arthas trace或SkyWalking查看调用链耗时分布 - 解决:对弱依赖做异步降级或增加超时重试策略

    第五,GC与JVM内部锁: - CPU不高但TPS低,可能出现频繁Young GC,导致"停顿但CPU不飙高" - 排查:jstat -gcutil <pid> 1000查看GC次数及耗时

    第六,操作系统限制: - 文件句柄数:ulimit -n默认为1024,高并发下socket连接不足

    Q3:如何定位到是String大Key导致的带宽问题?

    我使用了redis-cli --stat实时监控Redis节点的网络流量。观察到: - 每秒钟会有几次极高的入流量尖峰(如200Mbps),持续时间很短 - 结合--bigkeys找到那个2MB的Key,然后在业务高峰期用redis-cli --hotkeys观察该Key的访问频率 - 进一步用Arthas在应用端trace调用栈,找到执行redisTemplate.opsForValue().get(key)的代码行

    确认是记录"用户广告曝光历史"的模块。

    Q4:调优后广告平台的缓存命中率从60%回到95%,具体怎么做到的?

    大Key优化前,Redis内存90%+,触发allkeys-lru逐出策略,热数据被错误地当作"最近最少使用"而踢出,导致大量查询落到MySQL。

    优化措施: - 内存降到55%,不再逐出 - 为所有用户标签Key设置TTL(例如24小时滑动),且使用Hash结构,内存紧凑 - 增加本地缓存(Caffeine)二层缓存,极热数据在应用内存中命中,减少对Redis的调用

    优化效果: 缓存命中率从60%回升至94%~95%,数据库QPS下降70%。

    Q5:分布式压测中Master和Slave的带宽瓶颈如何避免?

    我采用了几个策略: - 设置JMeter Master的模式为StrippedBatch:mode=StrippedBatch,Slave只返回聚合后的指标,不返回完整的响应数据 - 将所有静态资源(CSV文件、jar包)提前分发到Slave节点,避免Master实时传输 - Master与Slave在同一内网万兆交换机下,网络延迟<0.1ms - 如果压测超过10k TPS,建议使用InfluxDB+Grafana实时收集Slave的指标,Master仅做调度

    Q6:如何保证压测数据不污染线上真实用户?

    这是压测的关键前提。我采取了以下措施: - 压测环境独立,使用影子库(表名加_test后缀)和隔离的Redis实例 - 预生成的userId落在专门的号段(如90000000~99999999),线上不存在 - 在压测请求的Header中增加X-Load-Test: true,业务代码埋点识别该标记,跳过非必要的埋点、日志打印、消息发送等副作用操作

    六、最终结果与闭环(30秒)

    通过这次压测和优化,我们取得了显著的效果:

    性能提升: - 响应时间:核心接口P99从800ms降至150ms以内 - 内存表现:Redis内存占用从90%降至稳定水位55% - 吞吐量:调优后稳定在5200 TPS,比优化前提升约8%

    业务价值: 成功支撑了2024年9月期间的高峰流量,限流阈值设定为5000 TPS,线上实现0报警。配合自动扩容机制,当限流触发频率过高时,云平台自动增加广告平台实例数。

    监控体系: 建立了完善的监控体系(Prometheus + Redis Exporter + Grafana),包括内存曲线、Key数量、逐出指标、大Key定期扫描、慢查询监控等,确保系统稳定运行。
