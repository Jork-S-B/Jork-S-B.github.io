    202409广告平台压测

    背景

    • 存量系统-广告平台出现预警：Redis内存占用持续上涨（突破 90%），接近爆满；且核心接口RT出现明显抖动（nginx-prometheus-exporter）。
    • 我首先根据报错日志排除连接数问题，随后排查了慢查询日志（阈值100ms），发现存在个别超过200ms的慢查询，以及某个大key（存储了广告位所有用户标签，value大小约2MB）。
    • 数据量级：
      • 用户标签库：约 5000 万用户画像数据。
      • 广告规则库：核心匹配规则约 500+ 条（MySQL 存储）。
    • 高峰流量：
      • DAU：12 万左右。
      • 日常峰值 TPS：约 1200 - 1500 TPS。
      • 期望容量：需支撑 3-5 倍冗余，即目标 TPS 需达到 5000 以上。

    初步定位：慢查询 + 大key 可能是导致Redis内存压力增大、接口耗时升高的原因。

    补充：数据来源

    压测接口为一个核心聚合接口（如 /api/v1/ad/match），其内部采用中台化分层调用模式。

    接口逻辑顺序（Pipeline）

    1. Request In：接收 userId 和 adSpaceId（广告位 ID）。
    2. Step 1：Redis 获取用户画像 (User Profile)
      • 通过 userId 查询标签。
      • 瓶颈点：由于标签存成了大 Key，每次 GET 都会导致网络 I/O 耗时增加，且 Redis 内部序列化/反序列化 2MB 数据非常耗CPU。
    3. Step 2：本地缓存读取广告配置 (Ad Config)
      • 使用 Caffeine 或 Guava Cache。
      • 逻辑：存储该广告位下挂载的候选广告列表，数据量小且稳定，几乎不消耗性能。
    4. Step 3：MySQL 获取精细化规则 (Ad Rules)
      • 逻辑：如果本地缓存失效，回源到 MySQL 查询特定的过滤规则（如：地域、年龄限制）。
    5. Step 4：核心逻辑匹配 (doJudge)
      • 逻辑：将 Step 1 的用户标签与 Step 3 的广告规则在内存中进行布尔运算（过滤、评分）。
      • 瓶颈点：如果标签过多，循环匹配的复杂度会上升。
    6. Step 5：外部风控 API 调用
      • 逻辑：最后一道防线。将匹配出的广告候选集发给风控系统，确认该用户近期是否点击过该广告，避免过度曝光。
      • 瓶颈点：这是全链路最慢的一环，通常需要 300ms-1s。

    压测策略与设计

    测试环境尝试复现

    • 配置：8C 16G * 2 节点（模拟生产 50% 的处理能力）。
    • 目的：初步探测代码性能拐点，验证调优方案的有效性。
    • 采用梯度加压：从500并发开始，每5分钟增加500，直到系统出现明显性能拐点（响应时间飙升、错误率上升）。

    灰度环境压测

    • 配置：16C 32G * 4 节点。
    • 集群规模：移动云集群部署，Redis 采用 8G 主从集群模式。
    • 凌晨低峰小并发加压压测。
    （单台）压测机配置与推荐线程并发数
    • 8C 16G：推荐线程数500-800
    • 16C 32G：推荐线程数1000-1500

    脚本设计

    开发提供的材料：接口文档、AESUtil.jar

    • 根据材料，以jmeter写压测脚本：

    1.使用BeanShell后置处理器对随机生成但有一定规则限制的号码进行加密，但运行报错：调用方法错误。

    2.使用jd-gui反编译jar包，初步分析：使用了lang3第三方工具包，正确方案：

    • 弃用 BeanShell，改用 JSR223 (Groovy)：BeanShell 性能极差，在高并发下会频繁触发 Full GC，拖慢压测机。Groovy 具有编译缓存机制，性能接近原生 Java。
    • 依赖管理：将 AESUtil.jar 和 commons-lang3.jar 放入 JMeter 的 lib/ext 目录，并重启 JMeter。

    3.为避免加密影响机器 CPU 消耗和 GC 压力，根据加解密的方法，生成了一批加密后的号码在csv文件中以参数化。

    4.分布式压测设置

    1. 设置堆内存配置（通常建议为物理内存的 50%）
    2. 设置内网ip，确保主机和从机可以互相识别和连接；master配置 Slave 列表。
    3. CSV 参数化文件分发到执行机
    4. master和slave关闭 SSL 认证（否则需要配置证书）：server.rmi.ssl.disable=true
    5. master默认没有响应结果返回，如有需要，则设置为Standard或StrippedBatch

    mode=Standard

    mode=StrippedBatch # 剥离响应数据只返回度量值，大幅降低 Master 的带宽压力。

    5.写shell脚本，创建以时间戳命名的独立结果目录、参数输入并执行分布式无界面压测、自动生成可视化 HTML 报告。

    #!/bin/bash

# --- 1. 默认值设定 ---
THREADS=10
DURATION=60
JMX_FILE="202409.jmx"

# --- 2. 参数解析 (sh run.sh -t 500 -d 600) ---
while getopts "t:d:f:" opt; do
  case $opt in
    t) THREADS=$OPTARG ;;    # 线程数
    d) DURATION=$OPTARG ;;   # 持续时间 (秒)
    f) JMX_FILE=$OPTARG ;;   # 指定 JMX 文件
    ?) echo "使用方法: sh run.sh -t <线程数> -d <持续时间> -f <脚本路径>"; exit 1 ;;
  esac
done

# --- 3. 路径与结果目录准备 ---
curr_time=$(date +%Y%m%d%H%M%S)
report_dir="./report_${curr_time}"
jtl_file="${report_dir}/output.jtl"
html_dir="${report_dir}/results"

mkdir -p "${html_dir}"

echo "=========================================================="
echo "执行模式: 分布式压测 (Remote Mode)"
echo "脚本文件: ${JMX_FILE}"
echo "并发线程: ${THREADS} (每个 Slave)"
echo "持续时间: ${DURATION} 秒"
echo "报告目录: ${report_dir}"
echo "=========================================================="

# --- 4. 执行 JMeter 分布式命令 ---
# -n: 非 GUI 模式
# -t: 指定脚本
# -l: 指定日志
# -e -o: 生成 HTML 报告
# -r: 启动远程节点 (remote_hosts)
# -X: 压测结束自动退出远程服务进程 (可选)
# -G: 全局属性，关键！用于将参数传递给所有 Slave

jmeter -n -t "${JMX_FILE}" \
  -l "${jtl_file}" \
  -e -o "${html_dir}" \
  -r \
  -Gtmp_thread="${THREADS}" \
  -Gtmp_duration="${DURATION}"

echo ">>> 压测任务完成。"
echo ">>> 请查看报告: ${html_dir}/index.html"

    瓶颈与优化

    总体包括：

    思路：先看DB（行锁/慢SQL/连接池），再看线程池（服务端+客户端），最后看OS/网络（端口/带宽）。

    大key

    • 定位过程：通过压测发现，随着并发量上升，内存几乎呈线性增长且不回落。我使用 redis-cli --bigkeys 分析，并配合 Arthas追踪业务逻辑代码。
    • 真因：发现系统在记录“用户广告曝光历史”时，使用了 String 结构且未设置合适的 TTL（过期时间）。在高并发下，每秒产生数万个过期时间极长的Key。当内存触达 maxmemory 后，触发了 allkeys-lru 频繁逐出有效热数据，导致缓存命中率从 95%跌至 60%，大量请求穿透到数据库，导致了接口变慢。
    • 解决建议：我建议开发将数据结构优化为 Hash（减少 Key 数量），并严格根据业务周期设置 TTL 滑动过期。优化后，压测显示内存水位稳定在 65%左右，不再持续上涨。

    补充

    • 定位深度：不仅仅是 bigkeys 发现 2MB。在面试时补充：“我通过 redis-cli --stat 观察到，在大 Key 读取时，内网带宽出现明显的入流量激增（Inbound Traffic），这确认了瓶颈在于 I/O 传输开销和应用端的反序列化压力。”
    • 优化原理：为什么改 Hash？“Hash 结构在字段较少时使用 ziplist 压缩列表，内存极其紧凑；且通过 HGET 仅取所需字段，将单次传输的数据量从 2MB 降到了 20KB 以内。”

    同步调用外部风控API导致的响应慢

    • 优化建议：非强关联的接口通过异步进行优化，如

      • 消息队列或事件驱动模型
      • 异步任务调度：使用线程池或协程异步执行
      • 通过重试、补偿等机制保证数据最终一致性

    最终选择了线程池异步任务的方式

    补充

    引入Sentinel (或 Hystrix) 进行降级。当风控 API 响应超过 500ms 时，自动熔断 10 秒，此时系统自动下发‘通用兜底广告’，优先保证核心业务不被外部依赖拖垮。

    最终结果与闭环

    • 响应时间：核心接口 P99 从 800ms 降至 150ms 以内。
    • 内存表现：Redis 内存占用从 90% 降至稳定水位 55%（通过 Hash 优化及过期策略）。
    • 业务价值：成功支撑了 202409 期间的高峰流量，限流阈值设定为 5000 TPS，线上 0 报警。

    📌 Q&A-压测过程

    Q：性能测试中如何评估并发用户数？

    如压测时要达到 1 万个 QPS，响应延迟最大 500 毫秒，怎么设置和计算需要多少台机器做并发？

    区分并发：1.同一时刻发起请求的用户数（集合点并发）2.在某个时间段内持续发压的虚拟用户数（线程并发）

    第一步：明确业务目标与指标口径，明确压测目的：容量评估、稳定性测试、还是摸高极限

    第二步：分类评估

    集合点并发

    定义：所有虚拟用户在集合点等待，同时发起请求，模拟瞬时高并发场景。

    适用场景：

    • 秒杀、抢购活动
    • 限时优惠活动
    • 系统启动瞬间流量
    • 定时任务并发执行

    评估方法：

    1.基于历史峰值数据 

    集合点并发数 = 历史峰值在线用户数 × 业务增长系数(1.5-2倍)
例如：去年双11秒杀峰值在线用户 5 万，今年预估增长 30%
集合点并发数 = 50000 × 1.3 = 65000 个虚拟用户

    2.基于业务预估 

    集合点并发数 = 活动总参与人数 × 参与率 × 同时抢购率
例如：活动宣传覆盖 100 万人，预计参与率 20%，同时抢购率 10%
集合点并发数 = 1000000 × 0.2 × 0.1 = 20000 个虚拟用户

    3.基于系统容量上限 

    集合点并发数 = 目标TPS × 平均响应时间 × 安全系数(1.2-1.5)
例如：目标支撑 10000 TPS（用户同时发起的业务请求数），平均响应时间 200ms
集合点并发数 = 10000 × 0.2 × 1.3 = 2600 个虚拟用户

    JMeter 实现方式：

    • 使用 Synchronizing Timer（集合点定时器）
    • 设置 Number of Simultaneous Users to Group by：集合的用户数
    • 设置 Timeout in milliseconds：超时时间（避免无限等待）

    某秒杀活动，预计 10 万人参与，历史数据显示 80% 用户在前 10 秒内抢购：

    • 集合点并发数 = 100000 × 0.8 = 80000 个虚拟用户
    • 分布式压测方案：使用 50 台压测机，每台 1600 线程
    • 设置集合点超时 5 秒，避免部分线程延迟影响测试结果

    线程并发

    定义：在某个时间段内持续发压的虚拟用户数，每个线程完成一次请求后立即发起下一次。

    适用场景：

    • 常规业务压测
    • 稳定性测试
    • 容量规划
    • 性能基准测试

    评估方法：

    使用 Little's Law（利特尔法则）：
并发用户数 = QPS × 平均响应时间

    说明：

    • QPS (Queries Per Second)：每秒查询数/请求数，表示系统吞吐量
    • TPS (Transactions Per Second)：每秒事务数，一个事务可能包含多个请求

    基础计算：

    • 目标：达到 10000 QPS，平均响应时间 500ms
    • 并发用户数 = 10000 × 0.5 = 5000 个虚拟用户

    考虑思考时间：

    并发用户数 = QPS × (平均响应时间 + 思考时间)

    实际案例（本次压测）：

    • 目标 TPS：5000（用户级别的完整业务流程）
    • 实际 QPS：约 15000-20000（系统内部的请求总数，包括 Redis、MySQL、风控 API）
    • 实际 P99 响应时间：150ms
    • 理论并发用户数 = 15000 × 0.15 = 2250 个（基于 QPS）
    • 实际设置：1000 个线程（考虑波动和余量，且部分内部调用有缓存优化）

    JMeter 实现方式：

    • 使用 Thread Group 配置线程数
    • 设置 Ramp-up Period：逐步加压时间（避免瞬间冲击）
    • 使用 Loop Controller 或 Duration Assertion 控制持续时间

    Ramp-up 时间建议：

    Ramp-up 时间 = 线程数 / 每秒启动线程数
一般每秒启动 5-10 个线程
例如：1000 线程，Ramp-up 设置 100-200 秒

    注意事项：

    • 压测机资源：单机线程数不超过推荐值（参考压测机配置）
    • 分布式压测：总线程数 = 单机线程数 × 压测机数量
    • 监控指标：TPS、响应时间、错误率、资源使用率
    • 持续时间：稳定性测试建议持续 1-2 小时，观察内存泄漏等问题

    第三步：结合案例

    若压测时要达到 1 万个 QPS，响应延迟最大 500 毫秒，怎么设置和计算需要多少台机器做并发？

    1. 参考公式：并发用户数 = QPS × 平均响应时间；即 并发用户数 = 10000 × 0.5 = 5000 个虚拟用户
    2. 基于现有压测机器配置，如
      • 8C 16G：推荐线程数500-800
      • 16C 32G：推荐线程数1000-1500
    3. 计算，向上取整，每个8C 16G压测机 700 线程，需要 7 台压测机。

    Note

    拓展：如何获取生产真实并发数据？

    1. 从生产 Nginx 导出 access.log，日志格式需包含 $time_local 或 $msec（毫秒级时间戳）。
    2. 统计秒级 QPS ：cat access.log | awk '{print $4}' | cut -b 2-20 | uniq -c 并实现 ELK 实时聚合，能看到曲线图，包括峰值时段、峰值请求量。 3.关键指标：取业务高峰时段（如晚 8~10 点、大促整点）的最大秒级请求数，这则是“集合点并发”的参考上限。

    Q：分布式压测中 Master 和 Slave 的带宽瓶颈如何避免？

    A：

    • 设置 JMeter Master 的模式为 StrippedBatch：mode=StrippedBatch。这样 Slave 只返回聚合后的指标（如响应时间分位值、错误计数），不返回完整的响应数据，大幅降低 Master 的带宽压力。

    • 将所有静态资源（CSV 文件、jar 包）提前分发到 Slave 节点，避免 Master 实时传输。

    • Master 与 Slave 在同一内网万兆交换机下，网络延迟 <0.1ms。

    • 如果压测超过 10k TPS，建议使用 InfluxDB + Grafana 实时收集 Slave 的指标，Master 仅做调度，不聚合数据。

    Q：假设压测时，TPS 上不去但 CPU 和内存都不高，你会从哪些方向排查？

    1.数据库层面（最可能）

    • 行锁竞争：当压测使用同一账户ID，数据库会串行化行锁，TPS上不去，而CPU/内存正常。

    • 排查：show engine innodb status查看锁等待；performance_schema统计锁等待时间。

    • 解决：改造为批量扣减+账户分片，或用Redis原子操作+异步落库。

    • 数据库连接池：连接池（如HikariCP）默认最大连接数可能只有10-20，压测时连接被占满，线程等待。

    • 排查：监控active connections、waiting threads。

    • 解决：适当增大连接池，但注意数据库自身连接数上限。

    • 还有慢查询

    2.服务端线程池配置

    • Tomcat/Undertow：maxThreads较小（如200），或acceptCount队列满了导致丢包。

    • 排查：查看服务端日志是否有“Thread pool exhausted”或拒绝连接。

    • 解决：增大maxThreads，同时结合压测并发数调整。

    3.压测机器及脚本问题

    • JMeter 非GUI模式：未启用-n -t -l，或Loop Controller未设置合理的pacing，导致线程空转。

    • 网络带宽：虽然CPU不高，但若响应体过大（如返回完整账户信息），千兆网卡可能打满。

    • 排查：iftop或nload观察压测机出网带宽。

    • 端口耗尽：短压测时，TIME_WAIT状态占满端口范围。

    • 排查：netstat -an | grep TIME_WAIT | wc -l。

    • 解决：调整net.ipv4.ip_local_port_range和tcp_tw_reuse。（或者改用长连接（Keep-Alive）可规避端口耗尽）

    • 还有压测机本身的配置，导致线程并发数上不去

    4.外部依赖/下游接口

    • 若扣减接口调用风控、积分等外部服务，且未设置合理超时+熔断，即使下游慢但未报错，也会导致线程阻塞。

    • 排查：用Arthas trace或SkyWalking查看调用链耗时分布。

    • 解决：对弱依赖做异步降级或增加超时重试策略。

    5.GC与JVM内部锁

    • CPU不高但TPS低，可能出现频繁Young GC（如每秒多次），导致“停顿但CPU不飙高”。

    • 排查：jstat -gcutil <pid> 1000 查看GC次数及耗时。

    • JVM内部同步：如Random、Logger、StringBuffer等隐式锁竞争。

    • 排查：jstack 抓取线程栈，统计BLOCKED状态的线程数。

    6.操作系统限制

    • 文件句柄数：ulimit -n默认为1024，高并发下socket连接不足。
    • 网卡队列溢出：ethtool -S eth0 | grep drop 查看是否有丢包。

    工具推荐

    • 行锁：MySQL · performance_schema.data_locks
    • 连接池：Druid监控页面、HikariCP的metrics
    • 线程栈分析：fastThread 或 jstack + 在线分析工具
    • 压测机端口：jmeter 改用长连接（Keep-Alive）可规避端口耗尽。

    Q：如何保证压测数据不污染线上真实用户？

    A：

    • 压测环境独立，使用影子库（表名加 _test 后缀）和隔离的 Redis 实例。

    • 预生成的 userId 落在专门的号段（如 90000000~99999999），线上不存在。

    • 在压测请求的 Header 中增加 X-Load-Test: true，业务代码埋点识别该标记，跳过非必要的埋点、日志打印、消息发送等副作用操作。

    Q：压测时如何判断“性能拐点”？

    A：

    • 响应时间：当并发增加，平均 RT 或 P95 RT 出现非线性增长（如从 100ms 突然跳到 300ms+）。

    • 吞吐量：TPS 不再随并发增加而增加，甚至下降。

    • 错误率：错误率突然从 0% 上升到 0.1% 或 1%。

    • 资源指标：CPU 使用率接近 85%~90%（尤其是 Redis 或应用容器），内存持续上涨不回收，磁盘 I/O 等待突增。

    例如本次压测中，10000 并发时 CPU 达到 85%，P95 RT 从 150ms 升至 1.3s，虽 TPS 仍高，但响应时间已不可接受，判定为拐点。

    Q：这次压测的限流阈值设定为 5000 TPS，具体怎么算出来的？

    A：基于系统最优 TPS（调优后稳定 5200 TPS）。

    • 留出 20% 安全余量：5200 × 0.8 ≈ 4160，仍需覆盖日常峰值 1500 × 3 = 4500。取整为 5000。

    • 限流方式：使用 Sentinel 的 QPS 限流模式，超过 5000 TPS 的请求直接返回 429 Too Many Requests 或排队等待。

    配合自动扩容：当限流触发频率过高时，云平台自动增加广告平台实例数。

    📌 Q&A-监控手段

    Q：你说内存从 90% 降到 55%，中间用了什么监控手段确保不再上涨？

    A：监控维度（采用 Prometheus + Redis Exporter + Grafana）：

    1. 内存曲线：used_memory / maxmemory，按小时/天查看趋势。优化前持续线性上升，优化后稳定在 55%。

    2. Key 数量：db0.keys，优化前因无 TTL 导致 Key 数量每天增长 10%+；优化后 TTL 生效，数量稳定。

    3. 逐出指标：evicted_keys，优化前每秒有大量逐出（因内存达到 maxmemory，allkeys-lru 踢出热数据）；优化后几乎为零。

    4. 大 Key 定期扫描：通过 --bigkeys 写入 cron 脚本，每日报告 >1MB 的 Key。

    5. 慢查询：slowlog get 100，监控是否有超过 100ms 的命令。优化前有 GET user:tag:xxx 耗时 200ms+；优化后无。

    📌 Q&A-问题定位与调优

    Q：如何定位到是 String 大 Key 导致的带宽问题？

    A：使用 redis-cli --stat 实时监控 Redis 节点的网络流量。观察到：

    • 每秒钟会有几次 极高的入流量尖峰（如 200Mbps），持续时间很短。

    • 结合 --bigkeys 找到那个 2MB 的 Key，然后在业务高峰期用 redis-cli --hotkeys（需要 LFU 策略）或直接通过 MONITOR 命令（生产慎用）观察该 Key 的访问频率。

    • 进一步用 Arthas 在应用端 trace 调用栈，找到执行 redisTemplate.opsForValue().get(key) 的代码行。确认是记录“用户广告曝光历史”的模块。

    Q：调优后广告平台的缓存命中率从 60% 回到 95%，具体怎么做到的？

    A：大 Key 优化前，Redis 内存 90%+，触发 allkeys-lru 逐出策略，热数据（如当日活跃用户的标签）被错误地当作“最近最少使用”而踢出，导致大量查询落到 MySQL。

    优化后：内存降到 55%，不再逐出。

    • 为所有用户标签 Key 设置 TTL（例如 24 小时滑动），且使用 Hash 结构，内存紧凑。

    • 增加本地缓存（Caffeine）二层缓存，极热数据（如广告规则）在应用内存中命中，减少对 Redis 的调用。

    效果：缓存命中率从 60% 回升至 94%~95%，数据库 QPS 下降 70%。
